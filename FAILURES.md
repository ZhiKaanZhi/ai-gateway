# Failures

Running taxonomy of where an agent, skill, or generated step produced bad output — and why. Evaluating the flow is the actual skill being practiced here, so log the failure, the root cause, and the fix. Thin and honest beats comprehensive.

### F1 — `pgvector-psycopg` skill: "a `list[float]` binds to `vector` directly" was wrong
**Surfaced:** Slice 1, the cache integration test — `psycopg.errors.UndefinedFunction: operator does not exist: vector <=> double precision[]`.
**Root cause:** The skill (rule 4) claimed a plain `list[float]` binds to `vector` once the type is registered. It does not: in pgvector ≥0.3 the psycopg dumper is registered only for `numpy.ndarray` and `pgvector.Vector`. A bare `list` falls through to psycopg's array dumper, so the value arrives as `double precision[]` and no `<=>` operator matches. `register_vector_async` fixes reads + the type OID, not list dumping.
**Fix:** Wrap at the bind boundary in `adapters/repository.py` — `Vector(embedding)` for the query and the stored vector — keeping the domain type `list[float]`. Corrected skill rule 4 and added a "Gotchas grows" entry so the next reader doesn't repeat it.
**Lesson:** Trust the skill for *where* the footguns are, but verify binding behaviour against the installed package version, not the prose.

### F2 — `docker-compose.yml`: PG18 volume mounted at `/var/lib/postgresql/data`
**Surfaced:** Slice 1, recreating the DB — container exited (1) with "in 18+, these Docker images store data in a major-version-specific directory… Counter to that, there appears to be PostgreSQL data in /var/lib/postgresql/data (unused mount/volume)".
**Root cause:** PG18 official images (which pgvector/pgvector:pg18 is built on) moved the data dir under a major-version subdir and expect the volume mounted at `/var/lib/postgresql`, not `…/data` (docker-library/postgres#1259). The bootstrap compose used the pre-18 `…/data` mount, so the container refused to start. It had been silently unhealthy; earlier gates only passed because no test touched the DB.
**Fix:** Mount the volume at `/var/lib/postgresql`. Cache is disposable (DECISIONS D2), so `docker compose down -v && up -d` to recreate cleanly.
**Lesson:** `docker compose up -d` returning success ≠ DB healthy. Check container health before trusting the DB leg of a gate.

### F3 — `pgvector-psycopg` skill: "under uvicorn it's fine" on Windows was wrong
**Surfaced:** Slice 1 manual round-trip — the app booted but every pool connection logged `Psycopg cannot use the 'ProactorEventLoop' to run in async mode`, so `/cache/*` couldn't reach the DB.
**Root cause:** Skill rule 9 said the Proactor/psycopg incompatibility only bites when "running asyncio directly," and that uvicorn was fine. Not on Windows: uvicorn's auto loop there is the ProactorEventLoop, which psycopg async rejects. The test suite hid this because conftest's `event_loop_policy` fixture already forced the selector loop — so tests were green while the real server was broken.
**Fix (two false starts before the real one):**
1. Set the policy at `main.py` import — **failed**: the uvicorn CLI creates its loop before importing the app, so the policy lands too late.
2. Set the policy in a launcher before `uvicorn.run` — **also failed**: uvicorn ≥0.36 passes a `loop_factory` to `asyncio.run` that hard-codes `ProactorEventLoop` on Windows (`uvicorn.loops.asyncio`), ignoring the policy entirely.
3. **Worked:** bypass uvicorn's loop selection — `asyncio.run(uvicorn.Server(config).serve(), loop_factory=asyncio.SelectorEventLoop)` in `gateway/__main__.py` (`python -m gateway`), guarded to `win32`. Corrected skill rule 9 to match.
**Lesson:** A green test suite can mask a runtime-only failure when the harness fixes the very thing production must fix itself. Do the manual round-trip — it caught what 8 passing tests didn't. And read the library's actual loop setup before assuming the asyncio policy is the lever.

### F4 — Slice 3 first pass: green gates, but a broken showpiece + real defects the reviewer caught
**Surfaced:** Slice 3 intent caching — `ruff`/`mypy --strict`/`pytest` were all green on the first pass, yet the `python-architecture-reviewer` (and then *running* the eval) found several genuine problems:
1. **`prompt_hash` declared nullable** (`text UNIQUE`, not `text NOT NULL UNIQUE`). The upsert keys on `ON CONFLICT (prompt_hash)`, but `NULL != NULL` in SQL — a null hash would silently insert a duplicate instead of collapsing it. The app never inserts null (the model requires it), so no test caught it; the DDL was a latent landmine.
2. **`evals/` imported `tests.conftest.FakeVerifier`** — eval code depending on the test package; fails outside a dev checkout. Tests passed because `tests/` was on the path.
3. **Private `_configure` imported across adapter modules** (with a `noqa`), plus two pools to the same DB. Collapsed to one shared pool, removing both smells.
4. **The eval headline was silently false.** The offline harness used Jaccard word-overlap as a similarity proxy. Word-overlap never reaches 0.97 for the surface-close cases, so the cosine-only baseline scored **0** false serves — making the showpiece print "baseline 0; gate 0", the *opposite* of the point (D30 is "baseline N; gate 0"). Every gate was green while the deliverable's one headline number was wrong. Fixed by labeling each case with the cosine a real embedder assigns (part of the adversarial design), so the harness reproduces the true contrast (baseline 4; gate 0).
**Root cause:** The quality gates check *type-safety and style*, not *semantic correctness of the artifact*. A green suite proved the code runs, not that the eval proves what it claims or that the schema enforces what the model assumes. Defect (4) only surfaced when the eval was actually *executed*, not asserted.
**Fix:** Addressed all four; re-ran `ruff`/`mypy`/`pytest` (46 pass) **and** the eval CLI to confirm "baseline 4 → gate 0".
**Lesson:** Green gates are necessary, not sufficient. For a *deliverable whose value is a number* (the eval), run it and read the number — don't trust a passing assertion that the headline holds. And run the architecture reviewer before declaring a slice done: it caught the packaging + schema defects the type checker structurally cannot.

### F5 — A passing adversarial eval never exercised the core mechanism; the centerpiece false-served transforms
**Surfaced:** Post-Slice-3, via an instrumented probe against the shipped gate (repo untouched). The intent gate served a transform-bound cached answer for a changed parameter: cached `"Translate 'hello' to Spanish"` → `"Hola."` was served for `"Translate 'goodbye' to Spanish"` (`serve=True`; expected refuse). With the `Verifier` wired to refuse (`0.0`) the gate **still served**, `verifier_calls=0` — the framed showpiece (the model confidence-gate) was never consulted.
**Cause:** Two surface-vs-semantic failures. (1) The binding check was a pure substring test (`param in answer`); a translation shares no letters with its input, so `"hello"` ∉ `"Hola."` → the answer was treated as unbound. Substring-contains ≠ value-dependence — the same class of mistake the slice exists to avoid (cosine ≠ confidence), repeated one level down. (2) The Verifier was gated on a confidence band that, for a single fresh candidate, was structurally unreachable (`base_confidence = 0.50 + 0.5·sim ≥ 0.85`), so it fired only on already-doubtful matches, never on a confident-looking transform — confidence and binding are orthogonal. The 8-case eval had **zero** cases that reached the Verifier: the four refuse cases all had answers that echoed their parameter (caught by substring) and the four serve cases were paramless. So the "cosine 4 → gate 0" headline was carried entirely by the substring heuristic + margin arithmetic — it demonstrated "a substring heuristic beats cosine," not "a model confidence-gate beats cosine."
**Lesson:** Green gates and a passing eval are necessary, not sufficient — an eval can pass while never exercising the mechanism it claims to prove. Adversarially probe the *centerpiece*: force the suspected path (here, wiring the Verifier to refuse and counting its calls) and confirm the behaviour actually depends on it. A labeled set must include cases that land on the path under test. Fixed in D32–D34; the eval now carries transform cases the Verifier must catch, plus a real-model hand-run (`evals/verify_live.py`) that validates the model isn't fooled (verification vs validation, cf. F4).
**Live-model result (2026-06-26, `gemma3:1b` via Ollama):** the validation half of the lesson, run for real. The four transform cases scored **0.20 / 0.25 / 0.20 / 0.40** — all below the 0.80 pass, so the *real* verifier (not just the mocked one) refuses the dangerous reuse. `"Hola."` for "goodbye" → 0.20: the model is genuinely not fooled. **But** the two value-independent serve-wins (`"…within 30 days."` for a different order; `"…Settings → Security → Reset."` for a different account) scored **0.50 / 0.30** — also below pass, so `gemma3:1b` would *refuse* reuses it could safely serve. The gate's precision bias means a weak verifier fails **safe** (wasteful, not dangerous), but the "win cosine can't take" needs a stronger verifier model or a tuned verify prompt than a 1B local model to actually land. Recorded, not papered over: the offline eval proves the gate *routes* correctly; the model is the limiting factor on recall, and that is a model/prompt problem, not a gate-logic problem.
**Live addendum (2026-06-26, one reframe attempt — `gemma3:1b`):** Reframed `_SYSTEM_PROMPT` from *relevance* to *answer-correctness* (credit an answer true for all cases; reject answers produced for a different input) and re-ran `evals/verify_live.py`. Six new scores — transforms (want < 0.80): `Hola.` **0.20**, `Lasting a very short time.` **0.25**, `56.` **0.80**, `≈ €9.20.` **0.87**; serve-wins (want ≥ 0.80): `…within 30 days.` **0.85**, `…Settings → Security → Reset.` **0.75**. **Outcome: not separated — and a precision regression.** The reframe pushed two transforms (arithmetic 0.80, convert 0.87) *up across* the 0.80 pass — they would now **false-serve** (the dangerous direction) — while a serve-win (password-reset 0.75) still under-scored. Lowest serve-win (0.75) sits below the highest transform (0.87), so no threshold separates the classes. Per the pre-agreed hard stop (one attempt, no second tuning pass), the prompt was **reverted** to the precision-safe relevance baseline and the recall gap **deferred** — see DECISIONS.md D35.
**Live addendum (2026-06-27, reframe reproduced — `gemma3:1b`):** Re-ran the same relevance→answer-correctness reframe live (Ollama up) to confirm the 2026-06-26 finding before deciding. Scores reproduced essentially identically — transforms (want < 0.80): `Hola.` **0.20**, `Lasting a very short time.` **0.25**, `56.` **0.80**, `≈ €9.20.` **0.87**; serve-wins (want ≥ 0.80): `…within 30 days.` **0.85**, `…Settings → Security → Reset.` **0.75**. **Outcome: not separated** (lowest serve-win 0.75 < highest transform 0.87; two transforms at/above the 0.80 pass = the same precision regression). Reverted again; deferral stands (D35). The finding is stable across runs, not run-to-run noise.
**Live addendum (2026-06-27, stronger model — `gemma3:4b`):** Tested the D35 "revisit with a stronger `GATEWAY_VERIFIER_MODEL`" lever (env override, no config commit), both prompt variants. *Relevance prompt* — transforms: `Hola.` 0.00, `short time` 0.00, `56.` 0.00, `≈ €9.20.` **0.95**; serve-wins: `…30 days.` **0.00**, `…Reset.` **0.30**. *Reframe prompt* — transforms: 0.00 / 0.00 / 0.00 / `≈ €9.20.` **1.00**; serve-wins: `…30 days.` **0.00**, `…Reset.` **1.00**. **Outcome: not separated on either prompt — and worse than 1b.** 4b is *confidently wrong in both dangerous directions*: it false-serves the currency-conversion transform (€9.20 reused for a different amount → 0.95 / 1.00) and false-refuses the universal return-policy serve-win (→ 0.00). Its scores are crisp (cluster at 0.00/1.00) but crisp-and-wrong; the reframe only rescues password-reset (→1.00). **Lesson:** model capacity alone is not the fix — a bigger general model is not automatically a better answer-correctness verifier for this task; the convert transform and the value-independent policy serve-win move the wrong way. The "stronger model" hypothesis in D35 is not borne out by 4b specifically.
**Live addendum (2026-06-27, `phi4-mini:latest` — incompatible output contract):** All six cases scored **0.00** on both prompts — but this is *not* the model's judgement, it is the precision-bias fallback (`ModelVerifier` returns 0.0 on any parse error). A raw probe confirmed phi4-mini ignores the bare-float contract and emits prose, e.g. for the return-policy serve-win: *"The cached answer provides a general statement… Relevance score = 0.1 (slightly relevant…)"* — `float(text)` cannot parse it, so every case falls to 0.0. **phi4-mini is not evaluable on this harness without a score-extraction parser**, and the strict bare-float contract is a deliberate design choice (prose-tolerant parsing invites rambling and non-determinism), so it stands. Even the buried 0.1 would be a serve-win false-refuse; no evidence phi4-mini separates the classes.
**Verdict across all tested models (2026-06-27):** For the verifier's actual job — refuse transforms (the safety-critical precision direction), serve value-independent answers (recall), and honor the bare-float contract — **`gemma3:1b` on the shipped relevance prompt is the best of the set, i.e. the configuration already shipped.** It is the only one that refused all four transforms with **zero false-serves** while honoring the contract; its weakness (under-scored recall, 0.50/0.30) fails *safe*. `gemma3:1b`+reframe regresses precision (two transforms ≥ 0.80); `gemma3:4b` (both prompts) confidently false-serves the convert transform (0.95/1.00) — the dangerous direction, disqualifying; `phi4-mini` is disqualified by the contract break. This independently validates the D35 deferral: the cheap shipped model is the safest available, and the recall gap is a property of the task's difficulty, not a model we simply failed to pick.

### F6 — Action intents (cancel/delete/transfer) are an unguarded reuse hole — RESOLVED (D43–D45)
**Surfaced:** While scoping the D32 fix. The intent tier treats every cached entry as a *question* whose answer can be reused. An **action** intent (`"Cancel order #1111"`) cannot be reused — it must actually run each time. A cached confirmation without the value (e.g. a bare `"Done."` or `"Your order has been cancelled."`) would, on `"Cancel order #2222"`, pass the Verifier (the text reads like a valid answer) and **confirm a cancellation that never executed**. The Verifier judges relevance, not whether the side effect happened, so it cannot guard this.
**Why not fixed cheaply now:** Detecting "this is an action" by a verb list (`cancel`/`delete`/`transfer`) is exactly the surface-proxy-for-a-semantic-property mistake this slice exists to kill ("contains the word cancel" ≠ "is an action"). A correct fix needs an execution seam: the provider performs the action and returns real confirmation, never reconstructed from cache. The live exposure is narrow — confirmations that echo the value are already caught by reject-fast (D32); only value-free confirmations are at risk — but it is real.
**Lesson / next step:** Name the hole rather than fake-guard it. Future slice: an action-executor port + provider call + confirmation semantics (a no-op/mock until a provider endpoint exists, but the contract must carry "it ran"). Caching is for questions; actions are out of the intent tier until then.
**Resolved (2026-06-28, D43–D45):** action-ness is read off the *model's reply* — a **tool call** is an action, plain text is a question (D43, no verb-list/no request flag — the same surface-proxy trap the lesson warned against). The reply carries a structured `tool_call` (D44); a guard in `RequestPipeline.process` returns early before the store block whenever `tool_call is not None`, so a tool-call reply is written to neither tier (D45). Never stored ⇒ never matched ⇒ never re-served: the hole is *structurally* impossible, not merely guarded — the same shape of closure as F5 → D32–D34. The originally-planned in-gateway `ActionExecutor` was dropped: the gateway is a proxy; the model emits the call and the *app* executes it.

### F7 — Slice 6 live run: the small backend model is tool-biased, defeating the FAQ-as-context demo
**Surfaced:** Slice 6 Track B (`-m live`, real `llama3.2:3b` via Ollama). The first cut of the live cache test asserted an FAQ question, sent *with the tools menu*, would be answered as text and then served from cache. It failed: given the menu, the model emitted a real `tool_call` — `get_order_status(order_id="<order status>")` — for "What is your return policy?". Probing further: the model calls `get_order_status` (often with an empty or junk `order_id`) for nearly every FAQ/policy question while the menu is attached, and *sometimes* emits the tool-call-shaped JSON as **plain text** (not via the `tool_calls` field), e.g. `{"name":"get_order_status","parameters":{"order_id":"returns_policy"}}` — which the gateway then caches as a normal answer.
**Root cause — not a gateway bug:** the gateway behaves exactly per D43/D45 (action-ness is read from the reply's *structure* — a real `tool_calls` field — never by surface-parsing text). A proper tool call is never cached; tool-call-shaped *text* is structurally a text answer and is cached. Defending against "text that looks like a tool call" would be the can't-tell-from-the-surface trap D43 deliberately avoids. The limiting factor is the model: a 3B local model with a tools menu present is over-eager to call a tool and under-eager to answer from context. This is the F4/F5 verification-vs-validation gap again — the deterministic Track A suite proves the gateway's *logic* (tool calls never cached; text served on re-ask), and only the live run reveals the *model* won't supply the cacheable text the FAQ demo wanted.
**Fix (test, not gateway):** the live cache test now (1) drops the tools menu — isolating the cache seam from the model's tool-bias; (2) asserts only that the *re-ask* is served from cache, not that the first ask missed (the DB is persistent across runs, so "first = miss" is not a live-safe assumption — Track A owns miss→store→hit deterministically); (3) self-skips if the model still answers with a tool call. The centerpiece live assertions pass: the model emits a real `cancel_order` tool call that the gateway parses and never caches, and the eshop executes it end-to-end (order flips to cancelled).
**Lesson:** a tools menu changes model behaviour — a tool-biased model will not exercise a text/cache path while the menu is attached, so test the cache seam without it. And a *persistent* cache makes "first call is a miss" an invalid live assertion; assert the durable post-condition (the re-ask is served), and keep miss→store→hit in the deterministic suite. The realistic product fix (don't attach tools for obvious FAQ turns, or a stronger/served model) is app-side and out of Slice 6's plumbing scope.
