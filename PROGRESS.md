# Progress

Milestone log and current state for the ai-gateway build. **Thin by design — the "where we are."**
The *why* behind choices lives in `DECISIONS.md`; agent/flow failures in `FAILURES.md`; the polished, outward-facing summary in `README.md`. Don't duplicate those here.

## Current state

Slice 6 done and green: the action path is **real, end to end**. A runnable eshop app (`src/eshop/`, in-memory orders + tool menu + FAQ) calls the gateway over HTTP; `ChatRequest`/`CompletionRequest` gained `tools` (opaque passthrough, D47) and `context` (D51); `openai_compat` forwards the menu, injects `context` as a system message, and parses `tool_calls` back out — first call only, `arguments` `json.loads`'d from Ollama's JSON string (D48). The app executes the tool the model picks against its store and returns a templated confirmation (D49); questions are answered from FAQ-as-`context` and the cache keys on the question alone, so they self-populate (D51). Actions are still never cached (the D45 guard). Backend default swapped to `llama3.2:3b` (gemma3:1b can't do tools); verifier stays `gemma3:1b` (D53). A console chat client (`eshop.cli`) drives it by hand. Two-track tests (D52): deterministic fakes in CI (gateway forwards + parses, `openai_compat` unit, eshop execute loop, FAQ self-populate) plus a self-skipping `-m live` real-model run. Builds on Slice 5's action seam (D43–D45). `ruff`, `mypy --strict`, and the full suite (72 tests; integration + `-m live` self-skip when Postgres/Ollama are down, so CI stays green offline) pass.

## Next step

**Conversation + messages slice (D50).** The request carries one user message today; the next slice adds multi-turn chat — and with it the cache-key policy it forces: when a request is several messages, what do you hash for the exact tier and embed for the semantic/intent tiers (the last message? a window? the whole transcript?). Until that's decided, multi-turn would silently break the matcher. Conversation memory is also the prerequisite for a model-written confirmation that can invite a follow-up (D49) — today's templated reply deliberately doesn't.

Parked: multi-tool calls (D48), model-written confirmations (D49), RAG-sourced `context` (D51); intent verifier value-independent recall — fails safe, see D35; revisit when a real workload justifies a stronger `GATEWAY_VERIFIER_MODEL`.
